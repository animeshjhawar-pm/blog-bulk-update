# Apply-to-S3 API blueprint

Draft for engineering review. Goal: design the production-grade "apply
regenerated image" pipeline in this repo, aligned with how
`gw-backend-stormbreaker` does it today.

Three intended scopes:

1. **Single image** — apply one image by its `image_id` UUID.
2. **Whole cluster** — apply every regenerated image within one cluster.
3. **Whole run** — apply every regenerated image across the whole run
   (multi-cluster).

Status as of writing: this repo has a primitive `applyOneHandler` +
`uploadBlogImage` that ships ONE size to three S3 keys and writes
nothing to the database. Stormbreaker's pipeline does considerably
more. The gaps drive every decision below.

---

## 1. How stormbreaker does it today

Mapped from `gw-backend-stormbreaker` + `gw-api-stormbreaker`. Cited
files + line ranges so engineering can verify.

### 1.1 Variant generation

Every image becomes **three WebP variants**:

| Width | S3 filename |
|---|---|
| 1080 px | `1080.webp` |
| 720 px  | `720.webp`  |
| 360 px  | `360.webp`  |

- Aspect ratio is **preserved** — no padding or cropping. A 1:1 source
  produces a 720×720, a 16:9 produces 720×405.
- Encoding: **lossy WebP, quality 80**.
- All three are produced in **one pass** by an **external AWS Lambda**
  (`sharp` under the hood — inferred from invocation pattern), not in
  the calling Python process.

Entry point: `services/image/image.py :: ImageService.convert_and_upload_as_webp()`
(lines 103–179). The Lambda payload lists the 3 variants and per-
variant `uploadOptions` (bucket, folder, filename); the Lambda
writes directly to S3 and returns the URLs.

### 1.2 S3 layout

Bucket name is environment-derived in `utils/constants.py`:

```python
S3_BUCKET_NAME = "gw-content-store" if env == "prod" else "gw-content-store-dev"
CDN_BASE_URL  = "https://file-host.link" if env == "prod" else "https://cdn-dev.gushwork.ai"
```

| Asset type | Key pattern |
|---|---|
| Blog inline (cover, thumbnail, infographic, internal, external) | `website/{staging_subdomain}/assets/blog-images/{cluster_id}/{image_hash}/{1080\|720\|360}.webp` |
| Generated (service, category H1/body, category-industry) | `website/{staging_subdomain}/assets/generated-images/{image_hash}/{1080\|720\|360}.webp` |
| Refined | `website/{staging_subdomain}/assets/refined-images/{image_hash}/{1080\|720\|360}.webp` |
| Operator-uploaded | `website/{staging_subdomain}/assets/uploaded-assets/{upload_id}` (no variants) |

Upload metadata: `Content-Type: image/webp`. Long-cache header
(`public, max-age=31536000, immutable`) is set on the fallback path
but not always on the primary one — worth standardising.

### 1.3 Database write

**Single row per image** in `media_registry`:

```python
{
  id:    uuid,                  # generated by RETURNING id
  p_id:  project_id,
  key:   "blog-images/{cluster_id}/{hash}",   # extracted from 1080 URL
  urls:  { "360": "...", "720": "...", "1080": "..." },  # JSONB
  d_at:  null
}
```

Insert: `services/postgres/stormbreaker/media_registry.py ::
bulk_insert_media_entries()` (batched `INSERT … RETURNING id`).

The returned UUID then gets attached as `image_id` to the consumer:
- **Blog inline images**: written into `clusters.page_info.blog_text.md`
  inside an `<Image imageId="UUID" alt="…" />` tag.
- **Service / category / cover-thumbnail**: attached as the `image_id`
  field on the matching record inside `page_info.images[]`,
  `page_info.fold_data.service_steps.images[]`, etc., matched by
  description.

Page-info update: `handlers/create_pages/update_images.py ::
save_images_to_page_info()` (lines 24–77). Walks the nested page_info
structure, finds the image record matching the description, mutates
`image_id` in place, calls `db.clusters.update_cluster_page_info(...)`.

**Not transactional** — media_registry insert and clusters update are
two separate SQL statements.

### 1.4 Replace-existing semantics

When regenerating an image that already exists:

- A **NEW S3 key** is written (new hash → new path).
- A **NEW `media_registry` row** is created (new UUID).
- The page_info is updated to reference the new UUID.
- **Old bytes stay in S3 forever.** No soft-delete, no cleanup.
- Old `media_registry` rows persist orphaned.

This means there's never a torn state where the page references a key
mid-upload — the swap is atomic at the page_info level.

### 1.5 API surface

There is **no single "publish this image" endpoint** in stormbreaker.
The flow is two-phase, async:

1. `POST /project/{project_id}/media/presigned-url` — returns a
   presigned S3 PUT URL the client uploads to directly.
2. `POST /project/{project_id}/media` — kicks off a Step Function
   that does the resize-to-3-variants + media_registry insert +
   page_info update asynchronously.

Both routes are in `gw-api-stormbreaker/serverless.yml`. Both gated by
`withAuthorization({ type: "project", permission: "can_edit_page",
extractProjectId: e => e.pathParameters.project_id })`. Auth is
Cognito JWT bearer (no custom signing keys).

For our purposes — internal admin tool, not customer-facing — the
two-phase pattern is overkill. We want a **single synchronous
endpoint** per scope (single / cluster / run) that does everything
inline.

### 1.6 Permissions

AWS IAM needed for the worker:

- `s3:PutObject` on `gw-content-store` / `gw-content-store-dev`.
- `s3:GetObject` on the same (read-back of variants if we ever
  verify, otherwise optional).
- No `s3:PutObjectAcl` needed — bucket policy controls reads via CDN.

Postgres: write access to `media_registry`, `clusters` (specifically
`page_info` column). The existing read-only `read_access_only` user we
use for the workspace render is **NOT sufficient** — we need a write
role. Engineering: confirm which role this app should run as.

## 2. What this repo does today (audit)

`src/s3.ts :: uploadBlogImage()`:

```ts
for (const size of APPLY_SIZES) {                       // ["1080","720","360"]
  const key = `website/${args.stagingSubdomain}/assets/blog-images/${args.clusterId}/${args.imageId}/${size}.webp`;
  await client().send(new PutObjectCommand({
    Bucket: bucket, Key: key, Body: body,               // ← same `body` for all 3 keys
    ContentType: "image/webp",
  }));
}
```

`src/web.ts :: applyOneHandler()` reads the parent run's CSV, locates
the row by `image_id`, calls `uploadBlogImage(...)` with the parent's
existing `staging_subdomain` + `cluster_id` + `image_id` + the new
Replicate URL.

### Concrete gaps vs stormbreaker

| # | What stormbreaker does | What this repo does |
|---|---|---|
| 1 | Generates 3 WebP variants at 1080/720/360 px | Uploads ONE WebP at native size to all 3 keys (the 720 and 360 files are 1080-px imposters) |
| 2 | Writes a `media_registry` row with `{ p_id, key, urls JSONB }` | Writes nothing to the DB |
| 3 | Updates `clusters.page_info` (blog_text MDX or images[].image_id) to reference the new UUID | Does not touch page_info — the published page keeps pointing at the OLD image_id |
| 4 | Each apply creates a NEW hash + UUID; old bytes orphaned | Reuses the OLD image_id, **overwriting** S3 in place (works as long as `image_id` is the canonical identifier — Sentinel/SpecGas hash IDs satisfy this; UUID-based assets do not) |
| 5 | Asset-type-aware S3 layout (blog-images vs generated-images vs refined-images) | Only the blog-inline pattern; cover, thumbnail, service H1/body, category-industry would write to the wrong S3 path |
| 6 | Cognito JWT + project-scoped `can_edit_page` permission | No auth at all |
| 7 | Async via Step Function | Synchronous, in-process |

### Practical impact today

The current `applyOneHandler` only works correctly for **Sentinel-/
SpecGas-style blog inline images** whose `image_id` is the canonical
S3 hash AND whose cluster doesn't track UUIDs in `page_info`. For
every other asset type the apply is silently wrong:

- **Service / category** images use UUID-based `page_info.images[].image_id`
  that maps to `generated-images/{hash}` in S3 — applying with our
  current path writes to `blog-images/...` which the page never reads.
- **Thumbnail** uses a different aspect ratio (3:2) and goes to
  `blog-images/{cluster_id}/{hash}` but its hash differs from the
  cover's — easily missed.
- **Cover image** stored via MDX UUID — the same UUID maps to a
  `blog-images/...` row in media_registry; overwriting the S3 key in
  place works ONLY if Cloudflare's CDN respects ETag changes (it
  doesn't by default — long-cache header means stale forever).

## 3. Proposed API surface

Three endpoints. All synchronous (acceptable: an apply takes 5–15 s,
well within Railway's request budget at our scale). All gated by an
internal bearer token (since this app already runs behind Railway
auth — engineering, confirm if Cognito is required instead).

### 3.1 Single image

```
POST /api/apply/image
Content-Type: application/json
Authorization: Bearer <token>

{
  "run_id":   "abc12345",
  "image_id": "5ba61e64-…"   // UUID from page_info, OR hash from S3 markdown
}
```

**Response (200):**
```json
{
  "ok": true,
  "image_id_old": "5ba61e64-…",
  "image_id_new": "f04c12ab-…",     // new media_registry UUID
  "key_prefix":   "blog-images/<cluster>/<new-hash>",
  "urls": { "360": "...", "720": "...", "1080": "..." },
  "page_info_updated": true,
  "elapsed_ms": 7421
}
```

### 3.2 Whole cluster

```
POST /api/apply/cluster
{ "run_id": "abc12345", "cluster_id": "b39b12ce-…" }
```

Same response shape per image inside an `applied: []` array, plus
`failed: []` for any individual failures. Best-effort: partial success
is reported, not rolled back.

### 3.3 Whole run

```
POST /api/apply/run
{ "run_id": "abc12345" }
```

Same shape as cluster, scoped to every cluster in the run.

### 3.4 Common backend pipeline

All three endpoints call into one internal `applyOne(runId, imageId)`
function. That function:

1. Reads the run's CSV row for `image_id` → gets
   `image_url_new`, `image_local_path`, `cluster_id`, `asset_type`,
   `client_slug`, `project_id`.
2. Resolves canonical S3 layout from `asset_type`:
   - cover / thumbnail / infographic / internal / external → `blog-images/{cluster_id}/{hash}/`
   - service_h1 / service_body / category_industry → `generated-images/{hash}/`
3. **Generates a new hash** (timestamp + 32 hex, matching stormbreaker's
   format), so the S3 path is new and we never overwrite.
4. Resizes to 3 widths (1080, 720, 360) — either via:
   - **Option A (recommended)**: call stormbreaker's Lambda directly (have engineering expose us the ARN + IAM access). Zero divergence.
   - **Option B**: use `sharp` in-process. ~30 MB binary, but Railway handles it.
   - **Option C**: precompute via Replicate already at 3 sizes — *not viable*, Replicate gives 1.
5. Uploads 3 WebPs to S3 with `Content-Type: image/webp` and
   `Cache-Control: public, max-age=31536000, immutable`.
6. Inserts into `media_registry` (project_id, key, urls JSONB) →
   returns new UUID.
7. Reads `clusters.page_info`, locates the matching image record by
   the OLD `image_id` (UUID or hash), rewrites its `image_id` to the
   new UUID. For blog inline images, this means rewriting the MDX
   `<Image imageId="…"/>` in `page_info.blog_text.md`.
8. Persists the updated page_info via `UPDATE clusters SET page_info
   = $1 WHERE id = $2`.
9. Writes the new image_id back into the parent CSV row so subsequent
   Download / Regenerate / Compare see the latest state (we already
   do this for Regenerate — same `updateParentCsvRow` helper).

Step 4 is the only meaningfully new piece of code. Steps 1–3, 5, 8–9
mostly exist or are trivial; step 6–7 needs a `db.ts` extension.

### 3.5 Open question: page_info MDX rewrite

For blog inline images the MDX inside `page_info.blog_text.md` looks
like:

```mdx
<Image imageId="5ba61e64-…" alt="Cross-section diagram of torn PTFE…" />
```

After apply we need to rewrite to the **new** UUID. Simple string
replace by old UUID is safe (UUIDs are globally unique). Alt text
stays. Engineering: confirm this is the right field to mutate and that
no other column needs to change (e.g. a separate publish-cache key).

## 4. Auth + permissions

**Decision needed from engineering.** Three options in increasing
strictness:

| Option | Description | Effort |
|---|---|---|
| A | Internal bearer token in `Authorization` header, hardcoded in env (`APPLY_API_TOKEN`). | 1 hour |
| B | Cognito JWT, validated against the same pool stormbreaker uses, requires `can_edit_page` on the project. | 1 day (need Cognito client config + permission service stub) |
| C | Mutual TLS via Railway private networking + service-to-service trust. | Project-level, ~1 week |

For an internal admin tool used by ~5 operators, **(A) is sufficient**
and keeps deploy story simple. If stormbreaker engineering insists on
parity, **(B)** is the upgrade path.

AWS IAM (regardless of which option above):

```json
{
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [ "s3:PutObject" ],
      "Resource": "arn:aws:s3:::gw-content-store/website/*"
    }
  ]
}
```

Add `s3:GetObject` only if we add a post-upload verification step.

Postgres role: needs `INSERT` on `media_registry`, `UPDATE` on
`clusters.page_info`. Engineering: name the role.

## 5. Rollout plan

1. **Phase 0 (this PR)** — land this blueprint, get review, lock the
   decisions in section 4 + 3.5.
2. **Phase 1** — implement the canonical pipeline. Wire `apply/image`
   first; reuse it from `apply/cluster` and `apply/run`. New file
   `src/apply.ts` with the pipeline; `db.ts` gains `insertMediaRegistry`
   + `updateClusterPageInfo`. `s3.ts` gains a `resizeAndUploadVariants`
   that wraps stormbreaker's Lambda (Option A) or `sharp` (Option B).
3. **Phase 2** — UI: per-card "Apply" already exists. Add
   "Apply cluster" and "Apply run" buttons in the publish view. Wire
   to the new endpoints. Concurrent applies fan out via `Promise.all`
   (already the pattern we use elsewhere).
4. **Phase 3** — add a dry-run flag (`?dry_run=1`) that returns the
   planned S3 keys + page_info diff without writing. Useful for
   review.
5. **Phase 4** — observability. Log per-image apply with
   `image_id_old`, `image_id_new`, `elapsed_ms` to stderr. Surface
   counts on the run page header.

## 6. Items needing engineering decision before code starts

1. **Variant generation path** — call stormbreaker's Lambda
   (recommended, zero drift) vs run `sharp` in-process (simpler, but
   duplicates resize logic). Need Lambda ARN + invoke permission if
   Lambda.
2. **Auth model** — bearer token (option A) vs Cognito (option B).
3. **Postgres write role** — which DB user, what grants.
4. **Page_info update for non-MDX clusters** — confirm
   `page_info.images[0].image_id`, `page_info.fold_data.service_steps.images[0].image_id`,
   `page_info.fold_data.service_description.images[0].image_id`, and
   `page_info.fold_data.industries.items[].image.image_id` are the
   four canonical fields the live page reads. Anything we're missing?
5. **Soft-delete of old `media_registry` rows** — keep stormbreaker
   behaviour (leak forever) or set `d_at` on the row we're replacing?
6. **CDN cache invalidation** — do we need to issue a CloudFront /
   Cloudflare purge after upload, or does the new key (hash differs)
   make this automatic?
7. **Sentinel-style image hashes** — confirm the
   `<timestamp>_<32-hex>` format we generate matches stormbreaker's so
   any downstream consumer that parses these doesn't break.

## 7. Open scope (deliberately deferred)

- Bulk regenerate-then-apply (one click, no intermediate review) — out
  of scope; operator review is the whole point of this tool.
- Cross-project apply — out of scope; each run is scoped to one
  project.
- Replicate model selection per asset type at apply time — already in
  place at generation time, doesn't change at apply.
- Rollback of an apply — out of scope; if a published image is bad,
  the operator can simply trigger a fresh regen + apply with the next
  attempt.
