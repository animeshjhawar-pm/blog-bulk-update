Blog Image Update

Bulk-regenerate cover, thumbnail, and inline images for published blog pages, with per-image apply-to-S3.
This Vercel deployment is a static landing page only. The actual app is a long-running Node server (subprocess spawn for runs, in-memory run state, SSE log streaming, filesystem I/O) — none of which fit Vercel's serverless model. Run it locally or host it on a platform that supports persistent Node processes (Render, Fly.io, Railway, an EC2 / Droplet, etc.).

Run it locally

  1. Clone animeshjhawar-pm/blog-bulk-update
  2. cp .env.example .env.local and fill in DATABASE_URL, FIRECRAWL_API_KEY, PORTKEY_API_KEY, REPLICATE_API_TOKEN, plus the AWS creds for S3 reads/writes.
  3. Install + start:
npm install
npm run web        # http://localhost:3000

Workflow

  1. Pick a client on the home page → land in its workspace with every published blog cluster listed.
  2. Tick whole clusters or open a row to pick individual images.
  3. (Optional) drop brand guidelines into the workspace panel — they get injected into every prompt.
  4. Dry run kicks off a fast mock pass that emits picsum URLs in seconds (no API spend).
  5. On the run page, each new image card has ↻ Regenerate (swap the image in place) and Apply to S3 (PUT to gw-content-store).

Why not Vercel

  1. Subprocess spawn — runs are launched via npx tsx src/cli.ts regen … in a child process. Vercel functions can't fork long-running children.
  2. In-memory state — the run registry (logs, SSE listeners, exit codes, paths) lives in a module-scope Map. Serverless invocations don't share memory.
  3. SSE — log streaming holds the response open until the run finishes. Vercel caps function duration; SSE that lasts minutes won't work.
  4. Filesystem — graphic-tokens/<slug>.json, out/<slug>-<utc>.csv, downloaded images. Vercel only has ephemeral /tmp.
A Next.js refactor + Postgres-backed run state + a queue (Inngest / QStash / SQS) for the regen workers would make it deployable on Vercel — that's the next iteration if you want to go that route.
